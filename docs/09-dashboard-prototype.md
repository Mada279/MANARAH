# 09 — نموذج لوحة Mishkat Dashboard

## الهدف

تحويل وثيقة Mishkat MVP من تصور نصي إلى نموذج بصري داخل الموقع الحالي، حتى يمكن مراجعة تجربة المنتج قبل بناء Backend وقاعدة بيانات.

## مكان النموذج في الكود

```txt
src/components/MishkatDashboard.tsx
```

وتم إدراجه داخل التطبيق بعد خارطة البناء وقبل قسم Manarah OS.

## ما يعرضه النموذج

النموذج الحالي Frontend فقط ويعرض بيانات تجريبية لـ:

- مؤسسة تجريبية.
- البرامج النشطة.
- المستفيدين.
- المتطوعين.
- ساعات التطوع.
- مؤشر الأثر العام.
- مؤشرات أثر تفصيلية.
- آخر الأنشطة.
- تقرير مختصر.

## لماذا هذه الخطوة مهمة؟

لأنها تنقل منارة من مستوى الرؤية إلى مستوى منتج قابل للنقاش:

```txt
رؤية
  ↓
وثيقة MVP
  ↓
Wireframe نصي
  ↓
Prototype بصري
  ↓
Backend + Database
```

## مكونات النموذج

### 1. Overview Cards

تعرض أهم مؤشرات المؤسسة:

- البرامج النشطة.
- عدد المستفيدين.
- عدد المتطوعين.
- مؤشر الأثر.

### 2. Active Programs

تعرض برامج المؤسسة مع:

- اسم البرنامج.
- الفريق المسؤول.
- عدد المستفيدين.
- عدد المتطوعين.
- نسبة التقدم.
- الحالة.

### 3. Impact Metrics

تعرض مؤشرات قابلة للقياس مثل:

- عدد المستفيدين.
- ساعات التطوع.
- اكتمال الأنشطة.
- رضا المستفيدين.

### 4. Activity Feed

يعرض آخر التحديثات التشغيلية.

### 5. Beneficiaries and Volunteers

جداول مبسطة لآخر المستفيدين والمتطوعين.

### 6. Impact Report Summary

ملخص تقرير أولي يمكن لاحقًا ربطه بمحرك التقارير.

## ما ليس موجودًا بعد

- لا توجد قاعدة بيانات.
- لا توجد API.
- لا توجد مصادقة.
- لا توجد صلاحيات فعلية.
- لا يوجد تصدير PDF حقيقي.
- لا توجد نماذج إضافة/تعديل بعد.

## الخطوة التالية

بعد مراجعة النموذج بصريًا، نبدأ في بناء النسخة التقنية:

1. إنشاء بنية Dashboard حقيقية.
2. إضافة صفحات Routes داخل التطبيق.
3. بناء Backend API.
4. إنشاء قاعدة البيانات وفق `05-data-model.md`.
5. ربط البيانات التجريبية بالـ API.
6. إضافة Auth وصلاحيات.

## معيار نجاح هذه المرحلة

تنجح هذه المرحلة إذا أصبح بإمكان صاحب المشروع أو مؤسسة تجريبية فهم شكل المنتج الأول خلال دقيقتين من النظر إلى النموذج.
